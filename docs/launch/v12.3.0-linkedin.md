# LinkedIn launch copy — cdxgen v12.3.0

## Option 1 — polished full post

We are getting ready to publish **cdxgen v12.3.0**, and this is one of the most meaningful releases we have shipped in a while.

Since `v12.2.1`, cdxgen has evolved in a big way across BOM generation, SPDX interoperability, upstream dependency review, runtime analysis, and release packaging.

The headline addition is **`cdx-audit`** — a new command that helps answer a practical security question:

> Which dependencies should we review first, and why?

Instead of only generating a BOM and stopping there, `cdx-audit` can start from one or more existing CycloneDX BOMs, resolve supported package URLs back to upstream source repositories, generate child SBOMs, and evaluate those sources with explainable rules and heuristics. The result is a more actionable review queue for AppSec and platform teams, with output available in **console**, **JSON**, and **SARIF**.

This release also adds **ATT&CK-aware** audit support, stronger GitHub Actions workflow analysis, and richer context for triage and escalation.

On the interoperability side, `v12.3.0` significantly improves **SPDX** workflows with a new **`cdx-convert`** command, better `cdxgen --format spdx` support, bundled **official SPDX 3.0.1** schema/model assets, and improved preservation of advanced CycloneDX semantics during export.

For runtime and operations-focused teams, we expanded cdxgen with **GTFOBins-backed container risk analysis**, stronger **OBOM runtime** rules, and better **LOLBAS / ATT&CK** context for Windows triage workflows.

We also broadened ecosystem and source intelligence with support for:
- **PEP 751 `pylock.toml`**
- purl source resolution for **GitHub**, **Bitbucket**, **Maven**, and **Composer**
- improved Maven/JAR metadata heuristics
- better CocoaPods lockfile handling
- proper npm handling for `devOptional` dependencies
- direct **Chrome extension** scanning and audit rules

The big theme in `v12.3.0` is simple:

**cdxgen is moving beyond “generate a BOM” toward “generate, convert, explain, and prioritize.”**

That means more value for:
- **developers** who need broader ecosystem support
- **AppSec teams** who need explainable dependency triage
- **SOC / IR analysts** who need runtime and host context
- **compliance teams** who need stronger SPDX workflows and validation confidence

A lot of work landed between `v12.2.1` and `v12.3.0`, including new commands, new rule packs, new helper modules, major documentation improvements, and better release assets.

Excited to share this release soon.
