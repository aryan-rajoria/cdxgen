# AI-Generated Code and Authorship Provenance

This document describes how to detect and record evidence of AI, LLM, and coding agent usage in developing, testing, reviewing, or triaging a project.

---

## Overview

CycloneDX code provenance detection allows you to analyze code repositories to collect signals of AI authorship or review. It stores these signals in namespaced CycloneDX custom properties (`cdx:ai:codegen:*`) and surfaces them as audit findings.

---

## Invocation

There are two entry points:

1. **BOM generation (`cdxgen`)** — opt in with the `ai-provenance` project type. The
   detected signals are written to the BOM document root `properties` array as `cdx:ai:codegen:*`:
   ```bash
   cdxgen -t ai-provenance -o bom.json .
   ```
2. **Auditing (`cdx-audit`)** — AI provenance detection is **enabled by default**. It runs
   whenever the `ai-provenance` rule category is active (all categories run by default):
   ```bash
   cdx-audit --bom bom.json --direct-bom-audit
   # scope to just AI provenance
   cdx-audit --bom bom.json --direct-bom-audit --categories ai-provenance
   # disable the detection
   cdx-audit --bom bom.json --direct-bom-audit --no-ai-provenance
   ```

If the audited BOM already carries `cdx:ai:codegen:*` properties (for example, generated with
`-t ai-provenance`), those are reused. Otherwise cdx-audit scans the target/working directory
and injects the properties before evaluating rules.

> Detection is intentionally **not** run on every `cdxgen` scan to keep ordinary SBOMs fast and
> free of noise. Use `-t ai-provenance` (or run `cdx-audit`) when you want the analysis.

---

## Detection Heuristics

The analysis scans five distinct channels:

### 1. Commit Trailers & Messages (Channel A)

Scans git commit messages and trailers for signatures left by AI assistants (e.g. `Co-authored-by: Claude`, `Generated by Devin`, `WindSurf`, `Aider`, etc.).

### 2. Bot / Machine Authors (Channel B)

Identifies commits authored by bots (e.g. `github-copilot[bot]`, `devin-ai[bot]`, `replit[bot]`, `coderabbitai[bot]`).

### 3. Configuration & Instruction Files (Channel C)

Checks for files indicating AI/assistant adoption, such as:

- `CLAUDE.md` / `.claude/settings*.json`
- `AGENTS.md` / `AGENT.md`
- `.github/copilot-instructions.md`
- `.cursorrules`
- `.windsurfrules`
- `.aider.conf.yml` / `.aider*`
- `.continue/config.yaml`
- `.clinerules`
- `.gito/config.toml`
- `agent.yaml`

### 4. CI / Review / Triage Workflows (Channel D)

Scans `.github/workflows/**` for references to known AI developer actions (e.g., `anthropics/claude-code-action`, `coderabbitai`, etc.).

### 5. Branch Naming (Channel E)

Checks if the current active branch name starts with patterns like `copilot/`, `cursor-`, `devin/`, or `codex-`.

Commit-history channels inspect only the most recent commits (default: 20). Recency is
sufficient to establish whether AI tooling is actively in use, and keeps the scan fast on large
repositories.

---

## Confidence Model

Signals are combined using a **noisy-OR** probability model:

$$\text{Confidence} = 1 - \prod (1 - c_i)$$

Where $c_i$ is the confidence score of each detected signal. The aggregate confidence is capped at `0.99` and categorized into bands:

- **High**: $\ge 0.80$
- **Medium**: $\ge 0.50$
- **Low**: $< 0.50$

---

## Output Properties

Properties are attached to the BOM document root `properties` array (not `metadata.properties`):

- `cdx:ai:codegen:detected`: `true` | `false`
- `cdx:ai:codegen:confidence`: Aggregate score (e.g. `0.95`)
- `cdx:ai:codegen:confidence:band`: `high` | `medium` | `low`
- `cdx:ai:codegen:tools`: Comma-separated list of detected tools
- `cdx:ai:codegen:phases`: Comma-separated list of development phases (`dev`, `review`, `ci`, etc.)
- `cdx:ai:codegen:signals:count`: Total signals found
- `cdx:ai:codegen:tool:<tool-name>`: Detailed evidence string for a specific tool (e.g., `confidence=0.98;phases=dev;channels=commit-trailer;evidence=1 commit(s)`)

### Git-ai derived detail properties

When [git-ai](https://github.com/git-ai-project/git-ai) notes are present and carry AI
attribution data, the following properties are additionally emitted:

- `cdx:ai:codegen:models` — comma-separated unique model names extracted from git-ai note sessions/prompts (e.g., `claude-sonnet-4-20250514,gpt-4o`).
- `cdx:ai:codegen:agents` — comma-separated unique agent tool names extracted from git-ai note sessions/prompts (e.g., `claude-code,cursor`).
- `cdx:ai:codegen:noteCount` — number of git-ai notes with AI attribution entries.
- `cdx:ai:codegen:sessionCount` — total number of AI sessions referenced across all git-ai notes.
- `cdx:ai:codegen:attributionCount` — total number of AI-attributed line+range entries across all git-ai notes.

### Pseudo-tools

Some signals indicate AI involvement that cannot be attributed to a specific product:

- `unattributed`: generic AI keywords in commit messages or AI hooks in `.pre-commit-config.yaml`.
- `multi-agent`: the cross-tool `AGENTS.md` / `AGENT.md` standard read by many agents.

These **pseudo-tools** still contribute to `cdx:ai:codegen:detected`, `:confidence`, `:phases`,
`:signals:count`, and the `:evidence:json` blob, but they are **excluded** from
`cdx:ai:codegen:tools` and do not get a `cdx:ai:codegen:tool:<tool-name>` property. This keeps the
tool inventory a clean, policy-friendly list of concrete products while preserving the underlying
evidence.

The `ai-provenance` rule category (`data/rules/ai-provenance.yaml`) turns the collected

| Rule         | Severity | Fires when                       | Focus                                                                            |
| ------------ | -------- | -------------------------------- | -------------------------------------------------------------------------------- |
| `AIPROV-001` | low      | `cdx:ai:codegen:detected = true` | AI/LLM/agent involvement; copyright & disclosure governance                      |
| `AIPROV-002` | medium   | confidence band is `high`        | High-confidence AI authorship; ownership, licensing, and human-authorship review |

## AI Human Oversight and Process Rigor

To move from _"was AI used?"_ (authorship) to _"was there meaningful human oversight?"_ (rigor),
cdxgen ships an AI human-oversight layer. When AI authorship is detected, it evaluates recent
git history — and, when a forge token is present, pull/merge request review data — to assign a
transparent rigor score (0.00–1.00) and surface governance gaps. It is neutral on whether AI is
good or bad; the deliverable is **evidence**, mapped to EU AI Act Art. 12/14, NIST AI RMF, and
ISO/IEC 42001.

The layer runs automatically alongside AI provenance (`-t ai-provenance` and `cdx-audit`).
`--no-ai-provenance` disables both layers.

### Data-source honesty

Local git **cannot** observe pull-request reviews or approvals. To avoid falsely reporting
"zero oversight", review-dependent metrics (`reviewCoverage`, `reviewLatencyVsSize`,
`selfMergeRate`, `verificationDebtRatio`) are reported as `unavailable` in git-only mode and only
become authoritative when a forge enricher supplies data. The channels that populated the score
are recorded in `cdx:ai:oversight:dataSources` (`git`, `git-ai-notes`, `github-api`,
`gitlab-api`).

### Configuration

- Set `GITHUB_TOKEN` or `GITLAB_TOKEN` (the same tokens cdxgen already uses for enrichment) to
  upgrade review metrics from `unavailable` to authoritative via the GitHub/GitLab REST APIs.
  This is opt-in and involves network access.
- `--git-ai-notes-ref <ref>` (cdx-audit) selects the git notes ref carrying
  [git-ai](https://github.com/git-ai-project/git-ai) line-level attribution (default:
  `refs/notes/ai`). Notes are not fetched by default; for CI run
  `git fetch origin "refs/notes/*:refs/notes/*"`.

### Metrics

The oversight collector emits the following `cdx:ai:oversight:*` properties. Numeric metrics use
fixed precision; review-dependent metrics render `unavailable` when no forge data is present:

- `verificationDebtRatio` — fraction of AI PRs/commits with no independent human review (headline; forge-only).
- `reviewCoverage` — fraction of AI PRs/commits with a verified independent human approval (forge-only).
- `reviewLatencyVsSize` — median review time ÷ diff size, a rubber-stamp indicator (forge-only).
- `selfMergeRate` — AI PRs merged/approved by their own author (forge-only).
- `batchSizeMedian` — median lines changed per AI commit (context).
- `revertHotfixRate` — reverts/hotfixes touching AI-authored work.
- `ciWeakeningEvents` — count of AI commits that delete tests or add quality-gate suppressions (`--no-verify`, `continue-on-error: true`, `|| true` in CI/build scripts).
- `signoffCoverage` — AI commits carrying a DCO `Signed-off-by` and/or verified signature.
- `codeownersCoverage` — fraction of AI-touched paths covered by CODEOWNERS (`unavailable` when no CODEOWNERS file).
- `aiLineRatio` — AI-authored lines ÷ total (precise from git-ai notes, else heuristic).
- `score` — aggregate 0.00–1.00 rigor score (transparent weighted mean over the available signals; `ciWeakeningEvents` applies an additional hard penalty).
- `band` — `strong` (≥ 0.75) / `moderate` (≥ 0.50) / `weak` (< 0.50).
- `dataSources` — channels that populated the score.
- `evidence:json` — structured per-signal evidence (sanitized JSON string) for a fully auditable, non-black-box score.

### Audit Rules

The `ai-oversight` category (`data/rules/ai-oversight.yaml`) evaluates process rigor and human oversight during `cdx-audit`:

| Rule        | Severity | Fires when                                            | Focus                                                |
| ----------- | -------- | ----------------------------------------------------- | ---------------------------------------------------- |
| `AIOVS-001` | high     | `verificationDebtRatio` ≥ 0.50 (forge data available) | High ratio of unreviewed AI code                     |
| `AIOVS-002` | high     | `ciWeakeningEvents` > 0                               | AI commits bypassing quality gates or deleting tests |
| `AIOVS-003` | medium   | `selfMergeRate` ≥ 0.50 (forge data available)         | AI code merged/approved by its author                |
| `AIOVS-004` | medium   | `reviewCoverage` < 0.50 (forge data available)        | Low share of peer-reviewed pull/merge requests       |
| `AIOVS-005` | low      | `codeownersCoverage` < 0.50 (CODEOWNERS present)      | Weak role accountability / AI attribution gaps       |
| `AIOVS-006` | medium   | overall band is `weak`                                | Poor human-in-the-loop oversight score               |

Review-dependent rules (`AIOVS-001`/`003`/`004`) only fire when a forge token supplies
authoritative data; in git-only mode those metrics are `unavailable` and the rules stay silent.

### Interpreting the oversight metrics

The score answers one question: **is AI-authored code being merged with a real, independent human
in the loop, or is it being rubber-stamped?** It is _not_ a measure of code correctness or quality
— a repository can ship subtly buggy code with excellent oversight, or perfect code with none. Read
the individual signals, not just the headline `score`/`band`.

#### What a strong (human-in-the-loop) workflow looks like

- AI-authored PRs are **reviewed and approved by someone other than the author** (and other than a
  bot) before merge → high `reviewCoverage`, low `verificationDebtRatio`, low `selfMergeRate`.
- Review time is **proportional to the size of the change** — large AI diffs get more scrutiny, not
  an instant approve → `reviewLatencyVsSize` is not near-zero for big batches.
- AI changes are **kept to reviewable batch sizes** → moderate `batchSizeMedian`.
- **Quality gates stay intact**: no deleted tests, no `--no-verify` / `continue-on-error: true` /
  `|| true` slipped into CI → `ciWeakeningEvents = 0`.
- Clear **accountability**: AI-touched paths are owned in `CODEOWNERS`, commits carry
  `Signed-off-by` and/or verified signatures → high `codeownersCoverage`, high `signoffCoverage`.
- Few **reverts/hotfixes** chasing AI changes → low `revertHotfixRate`.
- **Attribution is preserved** (git-ai notes present, `dataSources` includes `git-ai-notes`), so the
  AI line ratio is measured rather than guessed.

#### What a weak (rubber-stamping) workflow looks like

- AI PRs are **merged by their own author** or approved only by bots → high `selfMergeRate`, high
  `verificationDebtRatio`, low `reviewCoverage`.
- **Instant approvals on huge diffs** — hundreds of AI-authored lines approved in seconds →
  very low `reviewLatencyVsSize` with a large `batchSizeMedian`.
- **Gates are being weakened** in the same commits that add AI code → `ciWeakeningEvents > 0`.
- **No ownership or sign-off** on AI-touched paths → low `codeownersCoverage` / `signoffCoverage`.
- **Frequent reverts/hotfixes** shortly after AI merges → high `revertHotfixRate`.
- A high `aiLineRatio` **combined with** any of the above — a large fraction of the code is
  AI-authored _and_ the review signals are poor.

#### Factors to weigh when interpreting the numbers

| Signal / factor                    | Healthy direction                                   | Red flag                  | Why it matters                                                            | Caveats when reading it                                                                                            |
| ---------------------------------- | --------------------------------------------------- | ------------------------- | ------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------ | -------------------------- | -------------------------------------------------------------------------------------------------------- |
| `verificationDebtRatio` (headline) | low (→ 0)                                           | high (≥ 0.5)              | Share of AI work with **no** independent human review                     | Forge-only; `unavailable` in git-only mode means "not measured", **not** "zero debt"                               |
| `reviewCoverage`                   | high (→ 1)                                          | low (< 0.5)               | Share of AI PRs with a verified **independent** approval                  | Forge-only; a PR approved only by its author or a bot does **not** count                                           |
| `selfMergeRate`                    | low (→ 0)                                           | high (≥ 0.5)              | AI PRs merged/approved by their own author                                | Forge-only; solo maintainers legitimately self-merge — pair with team size context                                 |
| `reviewLatencyVsSize`              | proportionate                                       | ≈ 0 on large diffs        | Rubber-stamp detector: review time vs diff size                           | Fast reviews of _tiny_ changes are fine; judge relative to `batchSizeMedian`                                       |
| `batchSizeMedian`                  | small/moderate                                      | very large                | Large AI batches compress review capacity                                 | Context only; a big vendored/generated change can be legitimate                                                    |
| `ciWeakeningEvents`                | 0                                                   | > 0                       | Tests deleted or gates suppressed (`--no-verify`, `continue-on-error`, `  |                                                                                                                    | true`) in CI/build/scripts | Only counts CI/build/script/config files; deleting an obsolete test can be legitimate — inspect the diff |
| `signoffCoverage`                  | high                                                | low                       | DCO `Signed-off-by` / verified signatures = explicit human accountability | Many projects don't use DCO; low on its own is weak evidence                                                       |
| `codeownersCoverage`               | high                                                | low (CODEOWNERS exists)   | AI-touched paths have named owners                                        | `unavailable` when there is no `CODEOWNERS` file                                                                   |
| `revertHotfixRate`                 | low                                                 | high                      | Churn correcting AI changes hints at insufficient review                  | Recent-window heuristic; can be noisy on small samples                                                             |
| `aiLineRatio`                      | context                                             | high **with** weak review | How much of the code is AI-authored                                       | Not bad by itself — dangerous only when review signals are also poor                                               |
| `dataSources`                      | includes `git-ai-notes` + `github-api`/`gitlab-api` | `git` only                | More channels = higher-fidelity, authoritative score                      | `git`-only cannot see reviews, so review metrics read `unavailable` — set `GITHUB_TOKEN`/`GITLAB_TOKEN` to upgrade |

**Rules of thumb**

- Treat `git`-only results as **process-hygiene** evidence (CI weakening, sign-off, CODEOWNERS,
  reverts). Add a forge token before drawing conclusions about _review_.
- A `weak` band driven only by `unavailable` review metrics on a **work-in-progress branch** is
  expected — evaluate merged/`master` history, or after the reviewed PR merges.
- One high-severity signal (e.g. `ciWeakeningEvents > 0`) matters more than a mediocre aggregate
  `score`; always open `cdx:ai:oversight:evidence:json` for the raw per-signal numbers.
- The score measures **rigor, not correctness** — never treat a `strong` band as a guarantee the
  code is right, nor a `weak` band as proof it is wrong.

## Copyright and Intellectual Property Note

Under current U.S. Copyright Office guidance (_Copyright and Artificial Intelligence, Part 2:
Copyrightability_, 2025) and the D.C. Circuit's ruling in _Thaler v. Perlmutter_ (2025, cert.
denied 2026), copyright protection requires **human authorship**. Prompts alone do not give the
user authorship over an AI system's output. Consequently:

- **Code produced solely by AI may receive no implicit copyright protection** and could be freely
  used by others unless protected by another mechanism (licence, contract, or trade secret).
- **AI-assisted code can still be protected** where a human makes sufficiently creative
  contributions — substantial modification, or creative selection, coordination, and arrangement
  of AI output — assessed case by case.
- The practical takeaway is to **document human creative contributions** across the development
  lifecycle. The `cdx:ai:codegen:*` properties and `AIPROV-*` findings provide an auditable
  starting point for that record.

This is informational, not legal advice. Copyright treatment of AI-generated works varies by
jurisdiction and continues to evolve.

References: [U.S. Copyright Office — Copyright and AI, Part 2 (2025)](https://www.copyright.gov/ai/),
[Congressional Research Service — Generative AI and Copyright Law](https://www.congress.gov/crs-product/LSB10922).

## Privacy Note

When scanning commit history, author names and emails matching machine patterns are analyzed. Standard developers' emails/names are not targeted or stored, but repository commit messages and trailers are inspected locally.
